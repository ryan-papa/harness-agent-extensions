# rp-deck 디자인·전달력 규칙 (생성 SSOT)

장표 HTML을 생성할 때 반드시 따른다. 근거는 출처 표기.
비주얼 토큰·컴포넌트 마크업은 [`template.html`](template.html)이 SSOT — 클래스·CSS를 그대로 재사용하고 내용만 채운다.

## 1. 덱 구조 (전달 순서)

| 규칙 | 내용 | 근거 |
|------|------|------|
| 결론 먼저 | 각 섹션은 결론→근거 순. 빌드업 금지 | Minto Pyramid |
| SCQA 오프닝 | 표지 다음 도입부: 상황→문제→질문→답(권고) | Minto/SCQA |
| 피라미드 그룹핑 | 하위 근거는 상위 메시지로 수렴 (MECE) | Minto |
| 메시지 기반 | 목차·섹션은 주제 라벨이 아니라 주장으로 | Assertion-Evidence |
| 내러티브 | 데이터 나열이 아니라 긴장→해소의 흐름 | Duarte |

표준 흐름: 표지 → 목차(주장형) → [섹션구분 → 큰숫자/카드/대비/표/차트/다이어그램/단계/코드/공식] 반복 → 마무리 한 줄.

컴포넌트(마크업은 [`template.html`](template.html)): 표지·목차·섹션구분·큰숫자(KPI)·요약카드·2단대비·표·막대차트·**선차트(추세)**·**도넛(구성비)**·플로우·한줄강조·**단계(steps: key/bad/ok)**·**코드블록**·**공식강조**·용어각주(gloss)·**용어집(terms)**.

## 2. 슬라이드 한 장 원칙

| 규칙 | 내용 | 근거 |
|------|------|------|
| 헤드라인 = 완결 주장문 | 주제 라벨 금지. "매출이 40% 늘었어요"(O) / "3분기 매출"(X). **해요체 문장형** | Alley, 토스 UX writing |
| 한 장 = 한 메시지 | 주장 2개 이상이면 장을 나눔 | Duarte |
| 근거는 시각적으로 | 본문은 불릿 나열보다 차트·카드·다이어그램 | Alley |
| 3초 글랜스 테스트 | 3초 안에 요점이 잡히게 | Duarte/Reynolds |
| 여백 활용 | 핵심 하나를 여백으로 고립시켜 강조 | Reynolds |
| 시각 위계 | 장마다 지배 요소 1개, 크기·굵기·색·위치로 순위 | CLT |
| 일관성 | 그리드·타입스케일·색 역할을 전 장 고정 | CLT |

## 3. 데이터 시각화

| 규칙 | 내용 | 근거 |
|------|------|------|
| data-ink 최대화 | 그리드·테두리·3D·그림자·불필요한 색 제거 | Tufte |

| 직접 라벨 | 범례 대신 막대·선에 값 직접 표기 | UC Berkeley |

## 4. 인지 부하 감축

- 중복 효과 회피: 발표 스크립트 전문을 슬라이드에 그대로 넣지 않음 (Mayer)
- 청킹: 한 개념 = 한 단위로 쪼갬 (Mayer segmenting)
- 점진 공개: 복잡한 내용은 장을 나눠 단계로 (progressive disclosure)
- 신호화: 중요한 곳만 색·굵기로 큐, 장식 제거 (Mayer signaling)
- 근접성: 라벨은 대상 옆에 (split attention 회피)

## 5. 수치 가이드 (생성 시 강제)

| 항목 | 값 |
|------|-----|
| 헤드라인 | 완결 문장 1개, ≤~20자 권장 |
| 불릿 (쓸 경우) | 장당 ≤6개, 항목당 짧게 (6×6은 상한 heuristic) |
| 본문 최소 크기 | 18px+ (template 스케일 준수) |
| 대비 | 일반 텍스트 ≥4.5:1, 큰 텍스트 ≥3:1 (WCAG AA) |
| 폰트 | Pretendard 등 산세리프, 400/600/700만 사용 |
| KPI 카드 | 한 화면 ≤3 |
| 차트 시리즈/색 | ≤6 |

## 6. 토스 톤 & UX writing (본문 카피)

출처: Apps-in-Toss UX writing, toss.tech 8원칙.

| 규칙 | Don't → Do |
|------|-----------|
| 해요체 통일 | 모든 문장 해요체 |
| 문장형 > 명사형 | "송금 완료" → "송금했어요" |
| 능동형 > 수동형 | "됐어요" → "했어요" |
| 긍정형 > 부정형 | "이자 혜택 없음" → "이자를 더 받을 수 있어요" |
| 캐주얼 경어 | "확인하시겠어요?" → "확인할래요?" |
| 잡초 제거 | "앞으로 OOO을 하세요" → "OOO을 하세요" |
| 빈 문장 제거 | 제목을 본문에서 반복하지 않음 |
| 핵심만 | 이미 아는 정보는 생략 |
| 보편 단어 | 슬랭·밈·전문용어 금지 |

**주의**: 헤드라인은 "완결 주장문(해요체)". 명사 라벨("개요","결과")과 명사형 종결("송금 완료") 모두 금지.

## 7. 색 사용 규칙

- 인터랙션/강조: `--toss-blue`(#3182F6). 브랜드 블루(#0064FF)는 로고 등 브랜드 순간에만.
- 텍스트: 헤드라인 `--grey-900`, 본문 `--grey-700`, 캡션 `--grey-600`.
- 배경: 카드 white/`--grey-50`, 구분 `--grey-200` 헤어라인.
- 의미색만: 상승/긍정 `--positive`, 하락/부정 `--negative`. 색은 장식이 아니라 의미.
- 플랫 + 헤어라인 우선, 그림자는 실제 elevation에만 절제해 사용.

## 8. 용어 각주 (glossary)

독자가 뜻을 따로 찾지 않도록, 낯선 용어를 그 용어가 처음 나온 슬라이드 하단에 작게 설명한다.

| 규칙 | 내용 |
|------|------|
| 위치 | 용어가 처음 등장한 슬라이드 하단 (`.gloss`) |
| 선정 | 대상 독자에게 낯선 전문용어는 **웬만하면 다 설명**. 슬라이드당 **1~5개(필요하면 더)** |
| 약어 | 풀텍스트 병기 — `TTL(Time To Live): 자동 만료 시간` |
| 형식 | ` · `로 구분, 용어는 `<b>`, 뜻은 회색. 예: `쿼럼: 과반 정족수(N/2+1) · TTL(Time To Live): 자동 만료 시간` |
| 분량 | 뜻은 한 구절로 압축. **최대 3줄**까지 허용(넘으면 일부를 끝 용어집으로 이관) |
| 중복 금지 | 이미 설명한 용어는 뒤 슬라이드 각주에서 반복하지 않는다 |
| **끝 용어집** | 용어가 많은(대략 8개+) 기술 덱은 **마무리 직전에 "용어 사전" 슬라이드**를 둔다 — 본문에 나온 핵심 용어를 `.terms`로 모아 한눈에 |
| 생략 | 낯선 용어가 없으면 각주·용어집을 넣지 않는다 |

마크업은 [`template.html`](template.html)의 `.gloss`(하단 각주)·`.terms`(용어집 슬라이드) 참조.

## 9. 깊이와 분량 (얼마나 담을까)

전달력을 위해 줄이되, **이해에 필요한 내용까지 잘라내지 않는다.** 과도한 압축은 오히려 이해를 막는다.

| 규칙 | 내용 |
|------|------|
| 복잡도에 맞춰 | 슬라이드 수를 주제 난이도·분량에 맞게 늘린다. 얕은 주제는 적게, 깊은 주제는 충분히 |
| **개요 + 심화 2단** | 복잡한 메커니즘은 **개요 흐름 한 장(전체 그림) → 심화 여러 장(단계별 상세)**으로 항상 나눠 보여준다. 개요만 두고 끝내지 않는다 |
| **중간 단계 생략 금지** | 흐름을 "A → Z"로 뭉치지 말고 거쳐가는 **중간 컴포넌트·단계를 모두 드러낸다**(예: 인그레스 → 라우팅 → 서비스 디스커버리 → 엔드포인트 → 파드). 뭉뚱그린 화살표 하나로 넘기지 않는다 |
| 메커니즘은 단계로 | 동작·실패 과정은 `.steps`로 분해. 위험 단계는 `bad`(빨강), 해소 단계는 `ok`(초록), 핵심은 `key`(파랑) |
| 근거·수치·예시 | 주장에 근거·구체 수치·코드·시나리오를 붙여 납득시킨다 |
| 심화는 나눠서 | 한 장이 넘치면 개념→상세→시나리오로 장을 쪼갠다 (progressive disclosure) |
| 흔한 오해·엣지 | 자주 틀리는 지점·예외를 별도 장으로 짚는다 |
| 과소·과대 모두 지양 | 뻔한 내용으로 장 수만 늘리지 않는다. 각 장은 새 정보를 담는다 |
| **임의 상한 금지** | 20·25장 같은 숫자로 미리 자르지 않는다. 복잡한 주제는 30~35장+도 자연스럽다. 처음부터 복잡도에 맞춰 넉넉히 설계하고, 얇게 잡았다가 늘리지 않는다 |
| **낯선 도구 먼저 소개** | 처음 나오는 핵심 도구·서비스(예: Sentry)는 코드·설정으로 **쓰기 전에** "무엇인지·왜 쓰는지" 소개 장을 먼저 둔다. 각주만으로 대체하면 갑자기 튀어나와 어색해진다 |

핵심: "한 장 = 한 메시지"는 각 장을 얇게 만들라는 뜻이 아니라, 한 장이 한 가지를 **깊이** 다루라는 뜻.

## 10. 생성 전 자가 린트 (anti-pattern 체크)

- [ ] 텍스트 벽(문단/불릿>6/스크립트 전문) 없음
- [ ] 주제 라벨 헤드라인 없음 (전부 완결 주장문)
- [ ] 한 장에 메시지 2개 이상 없음
- [ ] 차트 정크(3D·그림자·그리드·의미없는 색) 없음
- [ ] 범례 의존 없음 (직접 라벨)
- [ ] 축 왜곡 없음 (막대 0 기준)
- [ ] 위계·타입스케일 장간 일관
- [ ] 대비 4.5:1 이상, 본문 18px 이상
- [ ] 예시 데이터에 각주, 실데이터 아니면 차트 남용 안 함
- [ ] 추세·구성비·비교 데이터는 적절한 시각 요소(선/도넛/막대/큰숫자)로 표현
- [ ] 시각 요소 옆에 해석·수치·용어 텍스트 병기 — 시각화가 설명을 대체하지 않음
- [ ] 카피 전부 해요체·문장형·긍정형
- [ ] 낯선 용어에 하단 각주(§8), 약어는 풀텍스트 병기, 각주 두 줄 이내
- [ ] 이해에 필요한 근거·수치·예시가 충분함(과도 압축 아님), 복잡한 메커니즘은 단계로 분해(§9)
- [ ] 복잡한 흐름은 개요 1장 + 심화 여러 장, 중간 컴포넌트·단계를 생략하지 않음(§9)
- [ ] 낯선 용어를 넉넉히 각주로 풀고, 용어 많은 덱은 끝에 용어집 슬라이드 포함(§8)
- [ ] 뻔한 내용으로 장 수만 늘리지 않음 — 각 장이 새 정보를 담음(§9)
- [ ] 슬라이드 수를 임의 상한(20·25장 등)으로 자르지 않고 복잡도에 맞게 확장(§9)
- [ ] 처음 등장하는 핵심 도구·서비스는 사용 전에 소개 장이 있음 — 각주만으로 대체하지 않음(§9)
- [ ] 개요 다이어그램이 본문의 모든 소스·목적지·경로를 누락 없이 담음(§9)

## 11. 회고 메모 (운영 학습)

|------|------|-----------|
