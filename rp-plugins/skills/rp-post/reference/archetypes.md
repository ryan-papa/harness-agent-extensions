# archetypes.md — 글의 원형

A1~A6 = 바이럴 원형(상위 25% 36개 전수 분류, 실측). K1~K3 = 사내 지식 공유 원형(지표가 아니라 전달력 목표, 존댓말). 소재 특성 × 목표로 2~3개를 골라 변형 초안을 만든다.

## 선정 가이드

| 소재가 이런 경우 | 1순위 | 2순위 |
|------|:---:|:---:|
| 성과·결과가 있는 경험담 | A2 | A1 |
| 진행 중인 고민·도전 | A1 | A3 |
| 극적인 사건·에피소드 | A3 | A1 |
| 만든 자료·도구·가이드 | A4 | A5 |
| 남의 사례·데이터 발견 | A5 | A6 |
| 업계 통념에 대한 반론 | A6 | A5 |
| **사내** 장애·삽질 해결 경험 | K1 | K2 |
| **사내** 새로 배운 개념·도구 | K2 | K1 |
| **사내** 기술 선택·아키텍처 결정 | K3 | K2 |

## A1. 동질감 소환형 — 답글 최강

- **공식**: 취약점 고백 → 구체적 상황 1~2개 → "비슷한 사람 있어?"
- **실측**: "뒤늦게 공부 시작한 사람 있어? …40후반에 내가 그런 사람이 되어 있네. 애 키우고 일도 하고 주경야독 중이야." (♥3,400 ↩944)
- **메커니즘**: 답글이 몰리면 알고리즘이 도달을 재부스트하는 복리 구조
- **주의**: 고백은 구체적이어야 함 (나이·상황·감정). 추상적 엄살은 역효과

## A2. 영수증 인증형 — 좋아요 최강

- **공식**: 반직관 성과 숫자(첫 줄) → 경력·자격 증명 → 실험·과정 예고 → 1/N 연재
- **실측**: "퇴사 후 한 달 동안 AI로 앱 30개를 만들었다. 나는 15년 경력 개발자다." (♥2,500 ⇄369 ↗743)
- **메커니즘**: 숫자가 신뢰와 호기심을 동시에 만든다. 연재 예고가 팔로우 전환
- **주의**: 숫자 없으면 이 유형을 쓰지 않는다. 소재에 숫자가 없는데 필요하면 사용자에게 확인

## A3. 미니 드라마형 — 몰입·체류

- **공식**: 평범한 일상 도입 → 짧은 문장 계단식 긴장 고조 → 반전 → 열린 질문
- **실측**: "이직 준비 6개월째였어. 점심시간에 화장실에서 몰래 전화 받고… 팀장이 갑자기 나 불러. …이직 포기해야 돼? 아니면 그냥 밀고 가야 해?" (♥1,200 ↩141)
- **메커니즘**: 300~600자 스토리 구간의 주력 유형 (중앙값 2배). 결말을 열어 답글 유도
- **주의**: 반전 없으면 일기가 된다. 반전 1개를 먼저 확보하고 역산 구성

## A4. 자격 선언 + 나눔형 — 답글:좋아요 1:1

- **공식**: 타깃 명시("~인 분들 위한") → 제작 배경·자격 → 수령 조건(답글/팔로우)
- **실측**: "바이브코딩 한번도 안해보신 분들 위한 〈웹사이트 만들기 가이드〉를 제작했습니다" (♥887 ↩912)
- **메커니즘**: 나눔 약속이 답글을 강제 — 답글이 좋아요를 넘는 유일한 유형
- **주의**: 실제로 나눠줄 산출물이 있어야 한다. 없으면 A5로 전환

## A5. 케이스 스터디·큐레이션형 — 공유·저장 최강

- **공식**: 팩트 숫자 나열(짧게 끊기) → "진짜 포인트는" 재해석 → 행동 촉구
- **실측**: "2명이 만든 게임이 1500만 장 팔림. 개발 기간 2개월… 단순 계산 매출 980억" (♥842 ↗248)
- **메커니즘**: 정보는 저장된다. 좋아요는 평범해도 리포스트·공유로 2차 도달
- **주의**: 재해석 없이 숫자만 나열하면 뉴스 요약이 된다. "그래서 무엇이 달라지나" 필수

## A6. 도발·논쟁형 — 도달 크지만 리스크 동반

- **공식**: 상대 진영 지목 → 연속 질문 공세 → (각오하고 쓸 것)
- **실측**: "바이브코딩으로 개발자 무시하는 사람들한테 한마디만 할게요. …혹시 자료구조가 뭔지는 아세요?" (♥500 ↩167 ⇄94)
- **메커니즘**: 진영이 갈리는 주제(바이브코딩 vs 전통 개발 등)에서 인용·답글 폭발
- **주의**: 브랜딩 리스크를 사용자에게 반드시 고지하고 선택받은 경우에만 사용

## K1. 트러블슈팅 회고형 — 사내에서 가장 검색되는 글

- **공식**: TL;DR(증상+원인+해결 3줄) → 증상·영향 범위 → 원인 분석 과정 → 해결 → 재발 방지·교훈
- **예시 훅**: "결론부터: OO 타임아웃의 원인은 커넥션 풀 고갈이었고, 설정 한 줄로 해결했습니다."
- **핵심**: 같은 문제를 겪을 미래의 동료가 검색으로 찾아 그대로 따라할 수 있게 — 버전·에러 메시지 원문·명령을 그대로 싣는다
- **주의**: 원인 규명 없이 "재시작하니 됐습니다"로 끝나면 지식이 아니라 일지

## K2. TIL·개념 정리형 — 배운 것을 팀의 자산으로

- **공식**: 왜 필요했나(문제의식) → 핵심 개념 요약(비유·그림 환영) → 우리 코드/업무 적용 예 → 적용 가이드·참고 링크
- **예시 훅**: "OO를 도입하며 공부한 내용을 정리해 공유합니다. 핵심만 보실 분은 표 하나만 보셔도 됩니다."
- **핵심**: 일반론 요약이 아니라 **우리 맥락**과 연결 — "우리 프로젝트에서는 이 부분에 해당합니다"
- **주의**: 공식 문서 번역·나열이 되지 않게. 내가 이해한 구조로 재구성

## K3. 의사결정 공유형 — 왜 그렇게 결정했는가 (ADR 스타일)

- **공식**: 결정 사항 한 줄 → 배경·제약 → 선택지 비교(표: 기준별 장단점) → 결정 근거 → 영향 범위·마이그레이션
- **예시 훅**: "OO에 A 대신 B를 사용하기로 결정했습니다. 비교 근거를 공유합니다."
- **핵심**: 반대 의견이 있었어도 기록 — 6개월 뒤 "왜 이렇게 했지?"에 답하는 문서
- **주의**: 결론만 통보하면 공유가 아니라 공지. 비교 과정이 본체

## 공통 규칙

| 규칙 | 내용 | 적용 |
|------|------|------|
| 훅 위치 | 첫 줄 20~40자. 첫 150자가 알고리즘 판단 구간 | A계열 |
| 두괄식 | 결론·TL;DR 먼저, 과정은 뒤에 | K계열 |
| 문장 | 짧게 끊는다. 느낌표 절제 (하위권 신호 −8%p) | 공통 |
| 숫자 | 형용사를 숫자로 치환. "많이" → "30개", "올랐다" → "4200→5600" | 공통 |
| 톤 | A계열 = 플랫폼 관례(쓰레드 반말 등) / K계열 = **존댓말** | — |
| 마무리 | A계열: 질문·나눔 약속·촉구 / K계열: 다음 액션·문의처 | — |
| 연재 | 500자 초과 스토리는 1/N 분할 — 1편 끝에 다음 편 예고 | A계열(쓰레드) |
