# platforms.md — 플랫폼 프로파일

기본 플랫폼은 쓰레드. 다른 플랫폼 선택 시 해당 프로파일의 제약·승자 공식으로 아키타입을 변주한다.
**증거 수준 표기**: 실측 = 자체 수집 데이터 / 리서치 = 외부 자료 기반 (초안 제시 때 사용자에게 한 줄 고지).

## threads (기본) — 실측

| 항목 | 값 |
|------|------|
| 글자 제한 | 500자/게시물 (초과 시 1/N 연재 설계) |
| 알고리즘 | 첫 150자로 판단. 답글이 돌면 도달 재부스트 |
| 승자 공식 | 1인칭 스토리 (+22%p), 300~600자 스토리 or ≤80자 스낵 |
| 패자 공식 | 리스트·가이드형 (−14%p, 단 저장·리포스트 목적이면 유효), 느낌표 연발 |
| 아키타입 | 6종 전부 적용 가능 |
| 금지 | 스팸성 해시태그 품앗이(#스하리 등) — 알고리즘 왜곡·기존 팔로워 이탈 |

## blog (벨로그·티스토리·미디엄 등) — 리서치

| 항목 | 값 |
|------|------|
| 유입 | 검색(SEO) 중심 — 피드와 승자 공식이 반대 |
| 승자 공식 | 튜토리얼·리스트형·문제해결 가이드 (실측 사례: WebRTC 튜토리얼 16만 뷰) |
| 길이 | 1,500~2,500단어 장문이 검색 상위 |
| 구조 | 제목(H1)에 키워드, 볼드 헤딩 + 불릿 + 코드 스니펫 + 이미지 (스캔 가능하게) |
| 타깃 | 초보 대상이 유리 (검색량 많음). 틈새 키워드로 좁히면 1페이지 노출 |
| 아키타입 | A5(케이스 스터디)·A2(영수증, 서사형 회고로) 적합. A1·A3은 도입부 훅으로만 |
| 배포 | 작성 = 절반. 커뮤니티 공유(HN·레딧·긱뉴스 등)로 트래픽 12배 사례 |

## naver-cafe — 실측 관찰 (표본 소규모)

| 항목 | 값 |
|------|------|
| 관찰 결과 | IT 카페 활동성 낮음 — 최대 카페(멤버 10.1만)도 주간 인기글 조회 214~489 |
| 문화 | 완성 콘텐츠보다 **도움 요청 질문**(조회 214·댓글 12)과 **업계 뉴스 반응**(조회 393)이 반응 획득 |

| 아키타입 | A1(질문형)·A5(뉴스+반응) 위주. A2 성과 인증은 자랑으로 읽힐 위험 |
| 시사점 | 브랜딩 목적이면 쓰레드가 압도적 우위. 카페는 타깃 커뮤니티 침투용 |

## company (사내 공유) — 목표가 다름: 지표가 아니라 전달력·재사용성

바이럴 지표(좋아요·답글)를 좇지 않는다. **내가 가진 지식을 동료가 이해하고 재사용할 수 있게** 정리하는 것이 목표. 체크리스트도 사내용(`checklist.md` §사내 공유)을 쓴다.

| 항목 | 값 |
|------|------|
| 톤 | **존댓말**, 건조하고 정확하게. 과장·어그로·이모지 연발 금지 |
| 아키타입 | K1(트러블슈팅 회고)·K2(TIL·개념 정리)·K3(의사결정 공유) 전용. A1~A6 바이럴 원형 사용 금지 |
| 구조 | 두괄식 — 결론·요약(TL;DR 3줄) 먼저, 과정은 뒤에 |
| 금지 | 시크릿·고객 데이터·인사 정보·내부 호스트/IP 등 민감 정보. 외부 공개 전제로 써도 안전한 수준 유지 |

**채널 2형** (변형 초안을 채널별로 하나씩 만드는 것을 권장):

| 채널 | 형식 |
|------|------|
| 슬랙 공유형 | ≤300자. 첫 줄 결론 → 핵심 2~3불릿 → 상세 링크(위키·PR·문서). 스크롤 없이 읽히게 |
| 위키 문서형 | 제한 없음. TL;DR → 배경·전제 → 본문(재현 가능한 상세: 버전·명령·코드) → 결론·다음 액션. 6개월 뒤 검색해서 읽어도 성립하게 |

## linkedin·기타 — 일반 원칙 (근거 약함)

| 항목 | 값 |
|------|------|
| 톤 | 전문성 + 1인칭 서사 결합. 도발(A6)은 지양 |
| 구조 | 첫 줄 훅 → 줄바꿈 잦게 → 교훈·인사이트로 수렴 |
| 아키타입 | A2·A5 적합 |
| 고지 | 실측 데이터 없음 — 초안 제시 때 명시 |

## 공통: 플랫폼 간 변환 규칙

| 방향 | 변환 |
|------|------|
| 쓰레드 → 블로그 | 스토리 훅을 서문으로, 본문은 목차·헤딩·코드 갖춘 장문으로 확장. 제목에 검색 키워드 |
| 블로그 → 쓰레드 | 결론·반전 하나만 추출해 1인칭 스토리로 재구성 + 원문 링크는 답글에 |
| 쓰레드 → 카페 | 성과 인증 톤 제거, 질문·정보 공유 톤으로 변환 |
| SNS → 사내 | 훅·성과 톤 제거, 존댓말 두괄식으로 재구성. 검증 안 된 주장은 근거 보강 또는 삭제 |
| 사내 → SNS | 민감 정보 제거 확인 후, 과정을 1인칭 스토리로 재구성 (A2·A3) |
