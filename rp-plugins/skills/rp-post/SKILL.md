# rp-post — 반응 나오는 글 초안 생성

정보를 나열한 글은 묻힌다. 실측 데이터가 증명한 구조(1인칭 스토리 훅 +22%p)로, 같은 소재를 터지는 글로 바꾼다.

## 레퍼런스 (반드시 먼저 Read)

| 파일 | 역할 |
|------|------|
| [`reference/patterns.md`](reference/patterns.md) | 실측 통계 SSOT — 상·하위 그룹 비교, 길이, 지표별 매트릭스, 데이터 갱신 절차 |
| [`reference/archetypes.md`](reference/archetypes.md) | 아키타입 — 바이럴 A1~A6 + 사내 지식공유 K1~K3 |
| [`reference/checklist.md`](reference/checklist.md) | 초안 검증 체크리스트 — 바이럴용·사내 공유용 각 7항목 |
| [`reference/platforms.md`](reference/platforms.md) | 플랫폼 프로파일 — 쓰레드(실측)·블로그·네이버카페·company(사내)·기타 |
| [`reference/review.md`](reference/review.md) | 독립 에이전트 병렬 리뷰 프로토콜 (Claude ∥ Codex 각 1회, 작성자 셀프리뷰 금지) |

## 입력 모드 (인자로 자동 판단)

| 인자 | 모드 | 동작 |
|------|------|------|
| 없음 | **컨텍스트** | 직전 대화·작업에서 소재 후보를 추출해 확인 후 진행 |
| 파일 경로 (존재하는 경로) | **파일** | 그 문서(회고·릴리스 노트·PRD 등)를 소재로 |
| 그 외 텍스트 | **소재** | 그 텍스트를 소재로. 사실·숫자가 부족하면 웹 검색으로 보강 |

## 워크플로우

1. **레퍼런스 로드** — `patterns.md` + `archetypes.md` + `checklist.md`를 Read. (매 실행마다)
2. **소재 확보** — 입력 모드 판별.
   - 컨텍스트 모드: 대화에서 소재 후보를 1~2줄로 요약해 "이 소재 맞아요?" 확인.
   - 소재에 검증 가능한 사실·숫자가 필요한데 없으면 WebSearch로 보강하고 출처를 함께 기록.
3. **부족 정보 질문 (최대 3개, 이미 알면 스킵)**
   - ① 목표: 좋아요(공감) / 답글(대화) / 리포스트·공유(도달) / 팔로워 전환 / **지식 공유(사내)**
   - ② 플랫폼: 쓰레드(기본) / 블로그 / 네이버카페 / **company(사내 — 목표가 지식 공유면 자동)** / 기타 → `platforms.md`의 해당 프로파일 Read
   - ③ 소재 디테일: 구체적 숫자·시점·감정 등 재료 (사내 모드는 재현 정보 — 버전·명령·에러 원문)
4. **아키타입 선정** — `소재 특성 × 목표`로 적합 아키타입 **2~3개** 선정. 선정 근거를 한 줄씩 제시.
   - SNS(A계열) 예: 성과 있는 경험담 → A2 영수증 인증형 + A1 동질감 소환형
   - 사내(K계열) 예: 장애 해결 경험 → K1 트러블슈팅 회고 (변형은 슬랙 공유형 + 위키 문서형으로)
   - ⛔ 사내 모드에 A계열(바이럴 훅·어그로) 사용 금지, SNS에 K계열 강제하지 않음
5. **초안 작성 (변형 2~3개)** — 각 변형은 해당 아키타입 공식을 그대로 따른다.
   - SNS: 훅은 첫 줄 20~40자, 자격·성과는 숫자로, 마무리는 목표 지표에 맞게 (질문/나눔 약속/행동 촉구). 톤은 플랫폼 관례.
   - 사내: 두괄식 TL;DR 먼저, **존댓말**, 근거·재현 정보 포함, 마무리는 다음 액션·문의처.
   - 플랫폼 프로파일의 길이·형식 제약 준수 (쓰레드 500자·1/N 연재, 슬랙 ≤300자+링크 등).
   - ⛔ 없는 사실·숫자를 지어내지 않는다. 소재에 없으면 `[숫자 채우기]` 플레이스홀더로 표시하고 사용자에게 확인.
6. **체크리스트 검증** — 변형마다 `checklist.md`의 **플랫폼에 맞는 체크리스트**(바이럴용/사내 공유용) 7항목 표(✓/△/✗ + 한줄 코멘트)를 첨부. 최저 항목이 ✗면 그 변형을 수정 후 재검증.
7. **선택·다듬기** — 사용자가 변형을 고르면 피드백 반영해 다듬는다 (반복 가능).
8. **독립 리뷰** — 다듬은 최종본을 [`review.md`](reference/review.md) 프로토콜로 검수. ⛔ 작성자 셀프리뷰 금지.
   - Claude 서브에이전트 ∥ Codex `codex exec` **병렬** 리뷰 각 1회 (같은 턴 동시 실행) → 취합 → high·med 반영, low 로그 → 종료 (수정 후 재리뷰 없음)
   - Codex 불가 시 1줄 기록 후 Claude 단독 진행 (비차단)
9. **마무리 출력** — 완성본을 복사해서 바로 붙일 수 있는 형태로 출력 + 리뷰 요약(지적/반영 건수) 1~2줄 첨부.
10. **(선택) 저장** — 사용자가 원하면 `<cwd>/docs/posts/YYYYMMDD_<슬러그>.md`로 저장 (초안 전 변형 + 최종본 + 체크리스트·리뷰 결과 포함).

## 출력 형식

```
## 변형 A — [아키타입명] (노리는 지표: OO)
[초안 본문]
— 체크리스트: [7항목 표]

## 변형 B — ...

추천: [어느 변형이 왜 목표 지표에 유리한지 한 줄]
```

## 절대 규칙

| 규칙 | 내용 |
|------|------|
| ⛔ 자동 게시 금지 | 이 스킬은 초안까지만. 플랫폼 게시는 항상 사용자가 직접 |
| ⛔ 셀프리뷰 금지 | 최종본 검수는 독립 에이전트([`review.md`](reference/review.md)) — 메인 에이전트의 자체 평가로 대체 불가 |
| ⛔ 사실 날조 금지 | 소재에 없는 성과·숫자·경력을 만들지 않는다. 플레이스홀더로 표시 |
| ⛔ 민감 정보 금지 | 실명·시크릿·고객 데이터를 초안에 포함하지 않는다. 사내 모드도 외부 유출 시 안전한 수준 유지 (사내용 체크리스트 #7) |
| 톤 분리 | SNS = 플랫폼 관례(쓰레드 반말 등) / 사내(company) = **존댓말** |
| 근거 우선 | 구조 선택의 근거는 항상 `patterns.md` 실측 데이터. 통념("짧게 써라" 등)과 충돌하면 데이터를 따른다 |
| 데이터 한계 고지 | 쓰레드 외 플랫폼 프로파일은 리서치 기반(실측 아님) — 초안 제시 때 이를 한 줄로 밝힌다 |
