# rp-codebase-recap

특정 레포를 **실제 코드 정독**으로 심층 발굴해 프로젝트 전모를 재구성한다.

## 철학

- ⛔ **레포 비종속(범용)**: 스택·도메인·구조는 레포마다 다르다. 특정 레포/언어/도메인에 특화·하드코딩 금지. 런타임에 실제 파일을 읽어 적응. 아래 예시는 방법일 뿐 기준 레포가 아니다.
- ⛔ **심층·전수 발굴(논문급)**: `git log` 제목 grep으로 끝내지 않는다. 유의미한 서브시스템을 **직접 Read**해 "무슨 문제를 어떤 구조로 풀었나"를 재구성한다. 기준은 **"모든 코드의 실질 내용을 담는다"**. 한 줄 요약·목록 나열로 끝나면 실패 — 다시 정독.
- **근거 필수**: 모든 주장·추정에 `파일:라인` 또는 커밋 해시 근거를 붙인다. 근거 없는 서술 금지.
- **최대 발굴 · 유저 필터**: 사실 후보를 최대한 꺼낸다. 무엇이 맞는지 판단은 유저. 불확실은 `(추정)`.
- **작성자 스코프**: 역할·기여도·담당영역·문제해결은 지정 작성자의 커밋/코드만. 개요·아키텍처·도메인기능은 전역.

## 트리거 · 입력 해석

- "이 레포/프로젝트 봐봐·파악해줘·분석해줘" / GitHub URL 공유 / `/rp-codebase-recap [대상] [작성자]`
- 명시 인자 강제 안 함. 맥락에서 추출:

| 값 | 해석 |
|----|------|
| **대상** | 로컬 경로·레포 이름(→ `repositories/` 등 탐색) 또는 GitHub URL |
| **작성자** | 유저 지정 우선. 미지정 시 최다 커밋자 추정+명시. **한 사람이 여러 신원**(이름/이메일)일 수 있음 → shortlog에서 후보 합산 후 확인 |

## 절차 (5-pass — 얕게 끝내면 실패)

### Pass 0 — 대상 확보
- **로컬**: 경로 확정(이름만 오면 탐색). **GitHub**: scratchpad에 `git clone --filter=blob:none` 후 분석, 종료 시 삭제. 비공개는 `gh`/git 인증 의존, 실패 시 사유 보고·중단.
- **비 git**: git 기반 섹션 스킵·명시, 코드 정독은 계속.

### Pass 1 — 광역 스캔 (지형 파악)
- 구조 `find -maxdepth 3 -type d`, 매니페스트 전수 Read, 규모/메트릭, 엔트리·라우터·빌드 설정.
- git 집계: 기여자·기간·churn.
- ⚠️ **출력 절단 주의**: 프록시/래퍼(RTK 등)가 `git log` 출력을 잘라 카운팅이 틀릴 수 있음. 커밋 수는 `git rev-list --count`·`git shortlog -sn`(집계형)으로, 목록형 `git log`가 필요하면 **원시 실행**(예: `rtk proxy git log …`)으로 재확인.

### Pass 2 — 핵심 서브시스템 선정 (정독 대상 리스트)
작성자 소유 코드 중 **유의미한 것**을 빠짐없이 후보화. 선정 신호:
- 대형 파일(LOC 상위) · 커밋 churn 상위 · 도메인 핵심 모듈 · 공통 인프라(플러그인·유틸·믹스인·API 계층·상태관리) · 아키텍처 경첩(엔트리·라우터·빌드).
- 목표는 **전수 커버**. "유의미한데 안 읽음"이 없어야 한다.

### Pass 3 — 심층 정독 (본체)
- Pass 2 리스트를 **실제 Read**. 대규모 레포는 **서브에이전트로 영역 분담 병렬 정독**(각자 한 영역 정독 후 구조화 반환)해 전수 커버.
- 각 모듈에서 추출: 책임 · 입출력/데이터플로우 · 상태·부수효과 · 의존 관계 · 엣지케이스 처리 · 눈에 띄는 알고리즘/자료구조 · 안티패턴/부채.

### Pass 4 — 문제해결 재구성 (케이스 스터디 — 논문급 핵심)
Pass 3 + git 히스토리(도입 시점·수정 churn·버그픽스·이슈번호)를 엮어 **케이스 스터디**를 쓴다. 유의미한 서브시스템 **건마다 1케이스**, 한 줄 요약 금지. 각 케이스 구조:

| 항목 | 내용 |
|------|------|
| 한마디로 | 전문용어 없이 한두 문장. "이게 무슨 기능이고 왜 어려웠나"를 비전공자도 알게 |
| 배경·문제 | 원래 무슨 문제였나 (코드 존재·구조·커밋에서 역추론, `추정`+근거) |
| 제약 | 기술/도메인/호환성 제약 |
| 접근·대안 | 택한 방식과 (드러나면) 버려진 대안 |
| 구현 핵심 | 실제 코드로 어떻게 — `파일:라인`, 핵심 로직·자료구조 |
| 트레이드오프·한계 | 비용·부채·놓친 케이스 |
| 결과·후속 | 이후 수정/버그픽스 churn, 안정화 여부 |

### Pass 5 — 종합·조립
아래 템플릿으로 터미널 출력. **저장 안 함.**

## 리포트 템플릿 (각 섹션 전수·근거 기반, 한 줄 요약 금지)

```markdown
# 코드베이스 발굴 리포트 — <레포명>
> 작성자 스코프: <X>  ·  대상: <경로/URL>  ·  git: <있음/없음>  ·  정독 파일 수: <N>

## 1. 프로젝트 개요 (전역)
- 서비스 정체 · 비즈니스 문제 · 타깃 사용자 · 규모(사용자/트래픽 단서) · 연동 시스템(백엔드·외부) · 근거

## 2. 기술 스택 (전역, 작성자 사용분 표시)
- 계층별 전수: 런타임·프레임워크·상태·라우팅·i18n·통신·빌드·린트·테스트 (버전)
- 선택 맥락·당시 대안·특이 선택 (추정+근거)

## 3. 아키텍처·구조 (전역)
- 엔트리/빌드 파이프라인 · 레이어 · 디렉토리 책임 맵 · 모듈 경계 · 상태관리 · 라우팅/가드
- 재사용 기제(플러그인·디렉티브·믹스인·공통 컴포넌트) · 렌더/성능 기법 · (텍스트 구조도)

## 4. 역할·기여도 (작성자 스코프)
- 커밋 수·점유율·기여자 N명 중 순위 · 활동 기간·밀도 · 국면(초기 구축/기능 확장/유지보수)

## 5. 담당 영역 (작성자 스코프, git log/blame)
- 소유 디렉토리·파일·컴포넌트 Top + blame 비중 + 어떤 계층을 책임졌는지 서술

## 6. 문제 해결 — 케이스 스터디 (작성자 스코프)  ← 리포트의 본체
- 유의미한 서브시스템마다 케이스 1건. 각 케이스는 **"한마디로"(쉬운 한두 문장) → 배경·문제 → 접근(일상 언어 설명, 용어 풀이) → 구현핵심(파일:라인) → 트레이드오프**.
- 전문용어는 그때그때 괄호로 풀이. 비전공자도 읽고 이해되게 자세히.

## 7. 정량 수치
- LOC·파일수·컴포넌트/모듈수·언어비율·엔트리·커밋수·기간·복잡도 분포(대형파일)·churn

## 8. 도메인 기능 목록 (전역)
- 전수 인벤토리: 카테고리 → 화면 → 기능. 화면/기능 수 집계

## 9. 리스크·기술부채
- 의존성 EOL/취약점 · 테스트 커버리지 · 복잡도 핫스팟 · 중복/결합도 · 접근성·보안 스멜 (각 근거)

## 10. 발굴 노트
- (추정) 항목·근거 요약 · 코드/git에서 미확보 항목(예: 정량 성과 지표) · 추가 확인 필요 · 정독 커버리지(안 읽은 영역 명시)
```

## 규칙

- ⛔ **저장 금지**: 터미널 출력만. 파일·볼트 기록 안 함.
- ⛔ **깊이 게이트**: 각 섹션이 육안 훑기 수준이면 실패 — 정독으로 되돌아간다. 특히 6번은 **케이스 스터디 형식 필수**, 커밋 제목 나열로 대체 금지.
- ⛔ **설명 눈높이(대학생·비전공자 이해 가능)**: 전문용어·약어·라이브러리·패턴 이름은 처음 나올 때 괄호로 쉬운 뜻을 풀이한다(예: "그리디(탐욕법 — 매 단계에서 당장 가장 좋아 보이는 걸 고르는 방식)"). 코드 인용 전에 **일상 언어로 "무엇을·왜"부터** 설명하고, 비유·예시로 직관을 먼저 준다. 압축해 짧게 쓰지 말고 풀어서 자세히. "그 분야를 몰라도 읽고 이해되면 통과, 아니면 실패."
- ⛔ **근거 없는 서술 금지**: 모든 항목에 `파일:라인` 또는 커밋 근거. 추정도 근거 첨부.
- ⛔ **작성자 스코프 위반 금지**: 4~6은 지정 작성자 코드만. 남의 기여 혼입 금지.
- **최대 발굴 · 커버리지 정직**: 후보는 다 꺼내되(추정 표기), 안 읽은 영역은 발굴 노트에 명시.
- **미화·각색 금지**: 특정 용도에 맞춰 미화·과장하지 말고 사실만. 순수 분석.
- ⛔ **민감정보**: 내부 호스트·IP·인증서 경로·개인 식별 정보·시크릿 노출 금지. 발견 시 리스크 섹션에 위치만 표기.
